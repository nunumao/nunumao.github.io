Maven (三) Maven 项目基础

第三章： Maven 项目基础

3.1 POM 文件 (Project Object Model)

正如我们之前所提到的，POM (Project Object Model) 是 Maven 项目的核心。 pom.xml 文件是 Maven 项目的灵魂，它是一个 XML 文件，描述了项目的 基本信息、构建配置、依赖关系 等等。

pom.xml 文件的作用:

  • 描述项目: POM 文件告诉 Maven 项目是什么，它有哪些依赖，如何构建，以及如何部署。
  • 配置构建: POM 文件定义了项目的构建过程，包括编译、测试、打包、部署等各个环节的配置。
  • 管理依赖: POM 文件声明了项目所依赖的外部库，Maven 会根据 POM 文件自动下载和管理这些依赖。
  • 插件配置: POM 文件可以配置 Maven 插件，扩展 Maven 的构建功能。

pom.xml 文件的基本结构:

一个基本的 pom.xml 文件通常包含以下元素：

主要元素解释:

  • <project>: pom.xml 文件的根元素。

    • xmlns 和 xsi:schemaLocation: XML 命名空间和 Schema 定义，用于验证 POM 文件的格式，通常不需要修改。
    • <modelVersion>4.0.0</modelVersion>: POM 模型的版本，固定值 4.0.0。

  • 项目坐标 (Maven Coordinates): 唯一标识一个 Maven 项目的三个核心元素。

    • <groupId>: 组织 ID，通常是公司或组织的域名反写 (例如 com.example, org.apache.maven). 用于区分项目所属的组织或团体。
    • <artifactId>: 项目名称，通常是项目的唯一标识符 (例如 my-maven-project, spring-core). 用于区分组织下的不同项目。
    • <version>: 版本号，表示项目的当前版本 (例如 1.0-SNAPSHOT, 1.0, 1.0-RELEASE). 用于区分项目的不同发布版本。

    Maven 坐标就像是项目的 “身份证”，通过 groupId:artifactId:version 可以唯一确定一个 Maven 项目及其版本。

  • <packaging>: Packaging 类型，指定项目的打包方式。 常用的 Packaging 类型包括：

    • jar: Java 归档文件，用于 Java 库或应用程序。 默认 Packaging 类型。
    • war: Web 应用程序归档文件，用于 Web 应用。
    • pom: POM 项目，用于项目聚合或继承的父模块，本身不包含可执行代码。
    • ear: 企业级应用程序归档文件，用于 Java EE 应用。
    • maven-plugin: Maven 插件项目，用于开发自定义 Maven 插件。

  • <name>: 项目名称，用于在 Maven 报告和文档中显示项目的名称，通常是用户友好的名称。

  • <url>: 项目 URL，指向项目的网站或文档地址。

  • <dependencies>: 依赖声明，用于声明项目所依赖的外部库。 我们会在第五章详细讲解依赖管理。

  • <build>: 构建配置，用于配置项目的构建过程，例如指定源代码目录、资源文件目录、插件配置等。 我们会在后续章节中逐步学习构建配置。

3.2 项目坐标 (Maven Coordinates) 详解

项目坐标 <groupId>, <artifactId>, <version> 是 Maven 的核心概念，理解项目坐标对于理解 Maven 的依赖管理和项目组织至关重要。

  • groupId (组织 ID):

    • 通常使用 域名反写 的形式，例如 com.example, org.apache.maven, org.springframework.
    • 代表项目所属的 组织、公司或团体。
    • groupId 的命名应该具有 全局唯一性，避免与其他组织的 groupId 冲突。
    • 例如，Apache Maven 项目的 groupId 是 org.apache.maven，Spring Framework 项目的 groupId 是 org.springframework.

  • artifactId (项目名称):

    • 代表组织下的 具体项目 的名称，例如 my-maven-project, spring-core, junit.
    • artifactId 的命名应该在 groupId 下具有 唯一性，区分组织下的不同项目。
    • 通常使用 小写字母、数字和连字符 组成。
    • 例如，Spring Framework 项目的 artifactId 包括 spring-core, spring-context, spring-webmvc 等。

  • version (版本号):

    • 代表项目的 版本，用于区分项目的不同发布版本。

    • 常见的版本号格式包括：

      • SNAPSHOT 版本: 表示 快照版本，开发中的不稳定版本。 例如 1.0-SNAPSHOT, 2.5-SNAPSHOT. Maven 会定期检查 SNAPSHOT 版本的更新。
      • RELEASE 版本: 表示 发布版本，稳定版本。 例如 1.0, 2.5, 3.0-RELEASE. 发布版本通常不会频繁更新。
      • 里程碑版本 (Milestone): 例如 1.0-M1, 1.0-M2.
      • RC 版本 (Release Candidate): 例如 1.0-RC1, 1.0-RC2.
      • Beta 版本: 例如 1.0-beta1, 1.0-beta2.
      • Alpha 版本: 例如 1.0-alpha1, 1.0-alpha2.

    • 版本号的命名约定和版本迭代策略是一个复杂的话题，需要根据项目的具体情况制定。 对于初学者，只需要了解 SNAPSHOT 版本和 RELEASE 版本的区别即可。

3.3 Packaging 类型详解

<packaging> 元素指定了 Maven 项目的打包方式，不同的 Packaging 类型会影响 Maven 的构建行为和最终的输出结果。

  • jar (默认):

    • 用于创建 Java 库 或 独立的 Java 应用程序。
    • 最终会生成 .jar 文件。
    • 默认的 Packaging 类型，如果 <packaging> 元素没有指定，则默认为 jar。

  • war (Web Application Archive):

    • 用于创建 Web 应用程序。
    • 最终会生成 .war 文件，可以部署到 Web 服务器 (例如 Tomcat, Jetty, WebLogic, WebSphere) 中运行。
    • war Packaging 的项目通常需要放在 Web 应用服务器的 webapp 目录下，并遵循 Web 应用的目录结构 (例如 WEB-INF 目录, index.jsp 等)。

  • pom (Project Object Model):

    • 用于创建 POM 项目，通常作为 父模块 或 项目聚合模块。
    • 最终会生成 .pom 文件，不包含可执行代码。
    • pom Packaging 的项目主要用于组织和管理其他 Maven 模块，例如定义公共的依赖和插件配置。

  • ear (Enterprise Archive):

    • 用于创建 企业级应用程序 (Java EE 应用)。
    • 最终会生成 .ear 文件，可以部署到 Java EE 应用服务器 (例如 WebLogic, WebSphere, JBoss EAP) 中运行。
    • ear 文件通常包含多个 war 文件和 jar 文件，以及 Java EE 应用的部署描述符。

  • maven-plugin (Maven Plugin):

    • 用于创建 Maven 插件。
    • 最终会生成 .jar 文件，可以作为 Maven 插件使用。
    • maven-plugin Packaging 的项目需要遵循 Maven 插件的开发规范。

如何选择 Packaging 类型:

  • 如果你的项目是 Java 库 或 独立的 Java 应用程序，选择 jar Packaging。
  • 如果你的项目是 Web 应用程序，选择 war Packaging。
  • 如果你的项目是 父模块 或 项目聚合模块，选择 pom Packaging。
  • 如果你的项目是 企业级应用程序 (Java EE 应用)，选择 ear Packaging。
  • 如果你的项目是 Maven 插件，选择 maven-plugin Packaging。

3.4 Maven 项目的标准目录结构

Maven 强制采用 标准化的项目目录结构，这是 Maven “约定优于配置” 原则的重要体现。 标准的目录结构使得 Maven 项目更易于理解和维护，也方便 Maven 自动化构建。

Maven 标准目录结构:

目录结构解释:

  • src/main/java: 存放项目的 Java 源代码。 按照包结构组织源代码。
  • src/main/resources: 存放项目的 主资源文件，例如配置文件 (.properties, .xml)、静态资源文件等。 这些资源文件会被复制到输出目录 target/classes 下。
  • src/main/webapp: 仅用于 war Packaging 的项目，存放 Web 应用相关的资源文件，例如 WEB-INF 目录、JSP 文件、HTML 文件、CSS 文件、JavaScript 文件等。
  • src/test/java: 存放项目的 单元测试代码。 按照包结构组织测试代码，测试代码的包结构应该与源代码的包结构保持一致。
  • src/test/resources: 存放项目的 测试资源文件，例如测试用的配置文件、测试数据等。 这些资源文件会被复制到输出目录 target/test-classes 下。
  • target: Maven 的 构建输出目录，Maven 会自动生成 target 目录，并将编译后的类文件、打包后的文件等输出到 target 目录下。 target 目录的内容可以通过 mvn clean 命令清理。

遵循 Maven 标准目录结构的好处:

  • 约定俗成: Maven 社区普遍遵循标准目录结构，方便开发者理解和维护项目。
  • 自动化构建: Maven 插件默认会按照标准目录结构查找源代码、资源文件和测试代码，实现自动化构建。
  • 提高效率: 减少配置工作，开发者可以专注于代码编写，而不是目录结构的组织。

总结第三章:

第三章我们深入学习了 Maven 项目的基础知识，包括 POM 文件的结构和内容、项目坐标、Packaging 类型和 Maven 项目的标准目录结构。 这些知识是理解 Maven 的基石。 掌握了这些基础知识，我们才能更好地使用 Maven 构建和管理项目。
